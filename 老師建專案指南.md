# 老師建專案指南：用 Obsidian + GitHub + Firebase 從零做一個工具

> 給看完 Claude基本功 EP05（GitHub）+ EP07（Obsidian）+ EP09.5（Firebase）的老師。
> 把**工作桌（Google 雲端硬碟）+ 三個家（GitHub / Firebase / Obsidian）** 組合起來，就能做出**會記住資料、能跨班使用、永遠在線，連你自己未來也不會忘記怎麼做的**小工具。
>
> **本指南只負責「思考順序 + 句型範本」，工具怎麼安裝請回去看對應集數的懶人包。**

### 🧭 本指南的基準線

本指南以 **「一台筆記型電腦、一位老師、一個學期的班級工具」** 為基準線撰寫。假設你會用到這個 **「1 桌 + 3 家」** 架構：

- 📋 **GDrive**（工作桌）：你實際編輯程式碼、CLAUDE.md 的地方

- **全校級工具 / 多位老師協作**：這份是入門；超過這個規模請見第 3.5 節「這套做法什麼時候不該用」

## 🗺️ 0.5 認知負擔地圖：哪些要懂、哪些只要說、哪些非你不可

看完全圖你可能會擔心「要學好多喔」。其實這份指南裡的內容分成三層，**只有第三層是真的要親手動**：

### 🧠 A 類：只要「懂」就好（概念，永遠不會自己寫）

這些你腦袋有模型就夠，語法由 AI 包辦：

- 前端 vs 後端、CRUD 四種操作、NoSQL 的 collection / document
- `onSnapshot` 訂閱、Serverless、Dev vs Prod、SSOT、Docs as Code
- 白名單安全規則、靜態 vs 動態資源該放哪個家

> 📌 這些**一次搞懂，終生受用**。沒有這些模型，你對 AI 下指令會打空砲；有了這些模型，一句中文就能動到對的地方。

### 💬 B 類：只要「說」就好（AI 完全代勞）

下面這些**全部用中文下指令、AI 做事**，你永遠不用真的會寫程式：

| 範圍 | AI 幫你做的事 |
|------|-------------|
| 🐙 GitHub | `git init`、commit、push、開 Pages、回退舊版本、修 GDrive Git 坑 |
| 🔥 Firebase | 建專案、建資料庫、寫 rules、部署、CRUD 全套、`onSnapshot` 程式碼、批次刪資料、匯出 JSON |
| 📘 Obsidian | 建同名資料夾、更新駕駛艙日誌、寫踩坑筆記、改 `CLAUDE.md` |
| 💻 前端 | 寫所有 HTML / CSS / JS、接 AI API、做 QR 產生器、預覽、debug |

第 5 節「句型範本庫」就是這類指令的**複製貼上模板**——根本不用動腦背，照抄改關鍵字就好。

### 🫳 C 類：只有「你」能做（AI 無法代勞）

無論 AI 多強都逃不掉的三類事：

1. **物理動作**：在 GDrive 開資料夾、`firebase login` 要你在 cmd 裡授權、用無痕視窗驗證「學生看到的版本」、用三個裝置測並發
2. **決策判斷**：想解決什麼問題（Q1）、選範式 A/B/C、判斷生命週期（Q6）、判斷「這件事該不該做」（第 3.5 節）
3. **責任把關**：上線前的檢查清單（第 7 節）、按下 push 讓學生看到的那一刻、決定要不要升級 Blaze 方案 / 找工程團隊

### 📊 三層的比例與力氣

| 類別 | 佔指南內容 | 要花的力氣 |
|------|----------|----------|
| 🧠 只要懂（A） | 約 35% | 中（一次搞懂就好） |
| 💬 只要說（B） | 約 55% | **低**（照句型貼即可） |
| 🫳 只有你做（C） | 約 10% | 高，但這**就是老師的本業**：判斷與責任 |

### 💡 關鍵洞察

這份指南 **90% 的內容**只需要「懂 + 說」，**不需要學會寫程式**。
**AI 把「手」的工作全包了，你只要用「腦」和「嘴」。**

真正的腦力應該花在 C：判斷該做什麼、該不該做、做完夠不夠好——**那才是老師無法被 AI 取代的價值。**

> 📌 如果你還在擔心「我不會寫程式怎麼用這份指南」，這張地圖就是答案：你根本不用寫，你只需要看得懂 A、說得出 B、在 C 的時候還是那個做決定的老師。

## 1. 工作桌 + 三個家的角色定位（工程師視角的核心概念）

**這一節是整份指南最關鍵的核心概念。搞懂它，後面任何專案你都會知道「這個東西該放哪」。**

### 核心洞察：每個專案有四種不同的資料

初學者最容易混淆的地方，是「所有東西都在 Claude 對話裡，不知道最後存到哪」。

其實每個專案都有四種**本質不同**的資料：

| 資料類型 | 特性 | 該住哪 | 為什麼 |
|---------|------|-------|--------|
| **進行中的編輯** | 你在寫、隨時會改 | **📋 GDrive 工作桌** | 自動跨電腦同步、Claude Code 直接讀寫 |
| **發佈的網頁／程式碼** | 凍結的快照、不會自己變 | **🐙 GitHub** | 需要版本控制、要公開給學生 |
| **執行中產生的資料** | 使用者互動產生、即時在變 | **🔥 Firebase** | 多人同時寫、要即時同步 |
| **思考與決策紀錄** | 你動腦產生、給未來的你看 | **📘 Obsidian** | 不該公開、不是程式 |

> 💡 **GDrive 跟 GitHub 內容會重疊**（程式碼進行版 vs 凍結版本），但定位不同：GDrive 是「**寫稿桌**」、GitHub 是「**出版社**」。

以下分別展開每個角色。

---

### 📋 GDrive 工作桌：你「實際工作」的地方

**一句話**：所有專案的工作目錄都放在 GDrive，Claude Code 直接編輯。

#### 工程師核心概念

**① 工作目錄（Working Directory）**
- 你的所有專案資料夾都在 `G:\我的雲端硬碟\` 底下
- Claude Code 桌面版打開的就是這個目錄
- 所有「進行中的編輯」都在這

**② 跨電腦同步（用 GDrive 取代本機 Documents）**
- 在學校編輯、回家自動同步、家裡電腦打開繼續做
- 不需要 git push（因為還沒到 GitHub 那階段）
- Google 自動處理同步衝突

**③ CLAUDE.md 的家**
- 專案藍圖檔（架構、技術決策、Do/Don't）
- 每次新對話 Claude 自動讀取
- 它跟程式碼住在一起，不是放別的地方

**④ 跟 GitHub 的關係：「寫稿桌 vs 出版社」**
- GDrive：你正在改的版本（半成品也在這）
- GitHub：完成的版本，push 後給世界看
- 你在 GDrive 寫 → `git push` 把定稿送到 GitHub → GitHub Pages 自動發佈

#### 初學者最常犯的錯

- ❌ 把專案開在 `C:\Users\xxx\Documents\` → 換電腦就讀不到
- ❌ 直接編輯 GitHub 網頁版的程式碼 → 跟你本機 GDrive 版不同步
- ❌ 把學生資料庫（會一直變動的東西）放 GDrive → 該放 Firebase
- ❌ OneDrive / Dropbox 也行，但別「同個專案放兩個雲端」（同步衝突）

#### 你要處理的事

- 每個新專案在 `G:\我的雲端硬碟\` 開資料夾
- 確認 GDrive 桌面版有同步狀態（綠勾 = 同步好）
- 所有編輯透過 Claude Code 做，少自己去動 GDrive 內檔案

---

### 🐙 GitHub 的角色：網頁與程式碼的「家」

**一句話**：你做出來的東西，放在這裡讓全世界看得到。

#### 工程師核心概念

**① 版本控制（Version Control）**
- 每次 `commit` = 一個存檔點
- 可以隨時回到過去某個版本
- 有人改壞也能還原
- 類比：遊戲裡的「存檔點」系統

**② 靜態資源（Static Assets）**
- HTML / CSS / JS / 圖片
- 上傳後**不會自己改變**
- 每個檔案都是「凍結的快照」
- 類比：報紙印出來就不會自己變內容

**③ 公開發佈（GitHub Pages）**
- 讓你的網頁有個公開網址
- 免費，不用租伺服器
- 只要 `push` 就自動更新
- 類比：把你做的紙本作品貼上公佈欄

**④ 協作紀錄（Commit History）**
- 誰、什麼時候、改了什麼，全部有紀錄
- 可以「怪罪」每一行程式碼的作者（`git blame`）
- 未來的你會感謝現在的你（寫好 commit 訊息）

#### 初學者最常犯的錯

- ❌ 把學生提交的資料寫進 HTML 裡，變動就 commit → 每次變動都要 push，完全不合理
- ❌ 把 API Key、密碼 commit 進去 → 公開 repo 所有人都看得到
- ❌ 改完東西忘了 push → 電腦壞掉就沒了
- ❌ commit 訊息都寫「更新」、「修改」→ 過三個月自己看不懂

#### 你要處理的事

- 每做完一個功能就 commit 一次
- `.gitignore` 要寫好（把 `.env`、`credentials.*`、`*.key` 加進去）
- commit 訊息要寫「做了什麼 + 為什麼」
- 完成一個里程碑就 push，然後用無痕視窗測 GitHub Pages 網址有沒有更新

---

### 🔥 Firebase 的角色：執行中資料的「家」

**一句話**：使用者跟你的網頁互動時，產生的資料都住在這裡。

#### 工程師核心概念

**① NoSQL 文件式資料庫（Firestore）**
- 不像 Excel 每列欄位都要一樣
- 每筆「文件（document）」可以有不同欄位（彈性）
- 用「集合（collection）」裝「文件」
- 類比：抽屜裡的卡片盒，每張卡片欄位可以略不同

**② 即時同步（Realtime / onSnapshot）**
- 資料一變動，所有訂閱的網頁**自動更新**
- 不用重新整理瀏覽器
- 這是 Firebase 的殺手鐧
- 類比：Google 文件多人共編，你打字別人的畫面就動

**③ 安全規則（Security Rules）**
- 決定誰能讀、誰能寫
- 預設應該是「白名單」：只開你要的，其他全擋
- 跟資料一起版控在 `firestore.rules`
- 類比：大樓門禁卡，哪張卡能進哪層樓

**④ 雲端執行環境（Serverless）**
- 你不用管伺服器
- Google 幫你管、備份、擴容
- 永遠不會關機
- 類比：你不用養雞，只要去雞蛋攤買蛋

#### Firebase 免費（Spark）方案的實際配額

「無限免費」是指**專案數無限**，但**每個專案**的資料庫操作有配額：

| 資源（Firestore）| 免費上限／天 | 換算成老師的使用情境 |
|---|---|---|
| 讀取（reads） | 50,000 次 | 30 人班級用 `onSnapshot` 即時顯示一堂課約 3,000～5,000 次，沒事 |
| 寫入（writes） | 20,000 次 | 30 人每人交 10 筆作業 = 300 次，綽綽有餘 |
| 刪除（deletes） | 20,000 次 | 老師手動清測試資料完全夠 |
| 網路傳輸 | 10 GB/月 | 純文字資料基本用不完；有圖片才要注意 |
| 儲存空間 | 1 GB | 一個班一學期文字資料遠低於此 |

**規模警示**：
- 班級工具（≤ 60 人）：基本用不完
- 全校工具（≤ 500 人）：一天 `onSnapshot` 連線數 × 資料變動次數要估一下
- 🔥 **大型研習 IRS（1000 人）**：1 次變動 × 1000 訂閱者 = 1000 reads；5 題就 5,000；若設計不當很容易破 50k 上限
  - 對策：Realtime Database（不是 Firestore）、或升級 Blaze 方案（按用量計費，有免費額度）
  - ⚠️ 「並發 100 萬」是 **Realtime Database** 的數字，**Firestore 沒有這麼大**，別搞混

> 💡 實務上，對國中小老師一個班級的應用，Spark 方案十幾年都用不完，不用擔心。但講「大型研習」時要老實說明這個差別。

#### 初學者最常犯的錯

- ❌ 把 Firestore 當 Excel 用，期待寫公式 → 該在 JS 或讓 Claude 分析
- ❌ Rules 選「測試模式」30 天後過期 → 應該直接寫白名單
- ❌ Rules 開全開（`allow read, write: if true;`）→ 全世界都能竄改你的資料
- ❌ 把學生真名寫進去 → 資料會經過 AI 做分析，個資風險

#### 你要處理的事

- 集合命名用英文小寫加底線（如 `wordcloud_words`）
- 每個集合都要有對應的 Rules 規則
- 用 Claude MCP 先測試「讀得到嗎 / 寫得進嗎 / Rules 擋得住惡意請求嗎」
- 去識別化：只存座號、班級代號，不存真名

---

### 📘 Obsidian 的角色：思考與決策的「家」

**一句話**：給人看的筆記，不是給程式讀的。

#### 工程師核心概念

**① 文件即程式（Docs as Code）**
- 全部用 Markdown 寫
- AI 最容易看懂的格式
- 純文字檔，沒有格式綁定
- 類比：跟程式碼享有同等「可版控、可搜尋、可分析」的待遇
- **深層意義**：**你的筆記和程式碼是平等的**——對 AI 來說都只是「可讀、可分析、可改的文字」。這也是為什麼 Claude Code 能直接讀你的 Obsidian、寫進你的 Obsidian、從你的 Obsidian 推導下一步該做什麼。Word 檔做不到，Notion 也做不到這麼直接，只有 Markdown 能。

**② 單一真理來源（Single Source of Truth，SSOT）**
- 進度、決策、踩坑只寫**一個地方**
- 避免「雙寫漂移」（兩處寫同件事，久了不一致）
- 本專案採用：`CLAUDE.md` 寫藍圖、Obsidian 寫日誌

**③ 連結思考（Graph / Backlinks）**
- `[[筆記名]]` 雙向連結
- 相關筆記互相串起來，像大腦神經元
- 未來搜尋「這件事我以前怎麼做的」很快
- 類比：維基百科每個詞都有內部連結

**④ 離線優先（Local-First）**
- 檔案就存在你電腦裡，不依賴任何雲端服務存活
- Obsidian 關掉、網路斷掉，筆記全都還在
- 即使只有一台筆電，你的「第二大腦」也完全掌握在自己手上

#### 初學者最常犯的錯

- ❌ 進度寫在 `CLAUDE.md`、`README.md`、Obsidian 三個地方 → 雙寫漂移
- ❌ 踩坑當下沒記，下次再踩一次 → 時間浪費在同個坑
- ❌ 把程式碼或設定放 Obsidian → 該放 GitHub
- ❌ 把學生真名或成績放 Obsidian → 該放 Firebase（且要去識別化）

#### 你要處理的事

- 每個專案在 vault 開一個**同名**資料夾
- 駕駛艙日誌集中在 `專案工作流程.md`（或同類檔案）
- 每次結束工作前更新「最近更動紀錄」
- 遇到新坑立刻寫進「踩坑筆記」

#### 🤔 Claude Code 有內建 Memory，為什麼還要用 Obsidian 駕駛艙？

Claude Code 其實有**內建 Memory 子系統**（Claude 自己寫給自己的工作筆記，存在 `~/.claude/projects/<專案>/memory/`）。功能上跟你的 Obsidian 駕駛艙有重疊——兩者都是「動態狀態 + 你糾正過的事 + 未完成的進度」。

那為什麼這份指南推薦你用 **Obsidian 駕駛艙當主要工作筆記**？

| 面向 | Claude 內建 Memory | 📘 Obsidian 駕駛艙 |
|------|------------------|---------------|
| 誰看得到 | 只 Claude 看（人類也能讀但不直覺） | **你跟 Claude 都看** |
| 跨工具讀取 | 鎖在 Claude Code 裡 | **任何能讀 Markdown 的工具都行** |
| 跨電腦 | 要靠 chezmoi 同步 `~/.claude/` 才有 | **你自己掌握**（vault 同步機制） |
| 工具中斷時 | Claude Code 不能用 = 讀不到 | **永遠讀得到**，純文字檔 |
| 寫入主動方 | Claude 自動寫（你不一定知道寫了什麼）| **你跟 Claude 共同編輯**（透明可控）|
| 連結與搜尋 | 純文字 | **雙向連結 + 全文搜尋**（Obsidian 強項）|

**一句話總結**：
- Claude 內建 Memory ＝「給 AI 看的工作日誌，人類順便也能讀」
- Obsidian 駕駛艙 ＝ **「給人類看的工作日誌，AI 順便也能讀」**

對老師來說，**「主控權在自己手上、跨工具能讀、Claude 收掉也不會消失」** 這三件事比 AI 自動化更重要。所以這份指南推薦把駕駛艙放在 Obsidian。

> 💡 兩者**不衝突**。如果你願意，可以兩者都用：Claude 自動寫 Memory、你週期性把重要的整理進 Obsidian。但對初學者來說，**先把 Obsidian 一個地方做好就夠了**——別一開始就維護兩套系統。

### 三個家怎麼合作？以「課堂文字雲」為例

按時間順序走一遍，看每個家各做什麼：

| 步驟 | 動作 | 動到哪個家 |
|------|------|-----------|
| 1 | 想到要做「即時文字雲」 | **Obsidian** — 開新筆記寫目的與設計 |
| 2 | 對 Claude 說「初始化專案」 | **GitHub** — 建 repo、push 初始 commit |
| 3 | 寫 `submit.html` 提交頁 | **GitHub**（檔案）+ **Obsidian**（紀錄為什麼這樣設計） |
| 4 | 對 Claude 說「建 Firestore 集合 wordcloud_words」 | **Firebase** — 建資料庫 + Rules |
| 5 | 寫 `display.html` 展示頁（用 onSnapshot） | **GitHub**（程式碼）+ **Firebase**（資料來源） |
| 6 | push 到 GitHub Pages | **GitHub** — 部署上線 |
| 7 | 學生掃 QR Code 提交 | **Firebase** — 收資料 |
| 8 | 公開展示頁即時顯示 | **Firebase → GitHub 上的網頁** |
| 9 | 老師用 Claude 查「最熱門的 5 個詞」 | **Firebase** — 分析 |
| 10 | 寫下今天的進度與踩坑 | **Obsidian** — 更新日誌 |

**關鍵觀察**：同一個「做文字雲」的工作，會動到三個家十次，每個家的角色都不一樣。

### 🎯 核心原則：每份資料只有一個家

這是整份指南最重要的一張表：

| 資料類型 | ✅ 正確的家 | ❌ 絕對不該去的地方 |
|------|---------|-------------------|
| **進行中的編輯**（程式碼草稿、CLAUDE.md） | 📋 **GDrive 工作桌** | 直接在 GitHub 網頁版改、放本機 Documents（不同步） |
| 發佈的網頁（HTML/CSS/JS） | 🐙 GitHub（從 GDrive push 來的版本） | 跳過 GDrive 直接編輯 GitHub |
| 學生提交 / 即時互動資料 | 🔥 Firebase | GitHub、Obsidian、GDrive |
| 你的決策、踩坑筆記、進度 | 📘 Obsidian | 程式碼註解、commit 訊息 |
| API Key / 密碼 | 本機 `.env`（絕不同步） | GitHub、Obsidian、GDrive（都會同步到雲端） |
| Firebase 公開設定（`firebaseConfig`） | 直接寫進 HTML 沒關係（這是公開資訊） | — |
| Firestore Security Rules | GitHub 上的 `firestore.rules` 檔 | 只在 Firebase Console 手動改（會跟本地版漂移） |

**口訣**：**在 📋 GDrive 工作桌做事 → 網頁去 🐙 GitHub、資料去 🔥 Firebase、想法去 📘 Obsidian。**

### 💡 五個工程師核心概念（不懂也能做，但懂了少走冤枉路）

#### ① 前端 vs 後端（Frontend vs Backend）

- **前端（Client-side）**：使用者瀏覽器執行的程式碼。HTML / CSS / JS 都是前端。
- **後端（Backend / Server-side）**：雲端伺服器執行。Firebase 扮演後端角色。
- **你的專案特徵**：Claude 幫你寫的大多是 **純前端 + Firebase**（serverless 架構），沒有自己架伺服器。所以工作輕鬆很多。

#### ② CRUD — 資料的四種基本操作

| 縮寫 | 全名 | 舉例 | Firestore MCP 工具 |
|-----|------|------|-----------------|
| C | **C**reate（新增） | 學生提交一筆心得 | `firestore_add_document` |
| R | **R**ead（讀取） | 展示頁列出所有心得 | `firestore_query_collection` |
| U | **U**pdate（更新） | 老師標記某筆「精選」 | `firestore_update_document` |
| D | **D**elete（刪除） | 老師刪除測試髒資料 | `firestore_delete_document` |

另外一個特殊的：**Subscribe（訂閱 / onSnapshot）** — 不是一次性讀取，而是「資料一變就通知我」，這是 Firebase 讓即時同步變簡單的關鍵。

#### ③ 開發環境 vs 生產環境（Dev / Prod）

- **開發環境**：你電腦上的 `G:\我的雲端硬碟\專案\`，Claude 改這裡
- **生產環境**：GitHub Pages 上學生真的在用的版本
- **流程**：先本機預覽 → 確認沒問題 → push 上 Pages
- ⚠️ 千萬不要在學生正在用的時候才 push 還沒測過的版本

#### ④ 為什麼 commit 訊息重要（比你想得還重要）

三個月後的你打開 GitHub，想找「上次某個功能是怎麼做的」：

- ❌ commit 訊息都是「更新」、「修改」→ 要一個個點開看
- ✅ 「加入即時派題功能，用 onSnapshot 訂閱 `quiz_assignments`」→ 一眼找到

這不是能力問題，是紀律問題。**寫好 commit 訊息是對未來自己的禮物。**

#### ⑤ Single Source of Truth（單一真理來源）

同一件事，**只在一個地方**寫。其他地方要參考，就用連結指過去。

| 事情 | 唯一真理來源 | 其他地方該怎麼做 |
|-----|------------|---------------|
| 專案藍圖（為什麼這樣設計） | 專案資料夾的 `CLAUDE.md` | 引用、不重寫 |
| Firestore 規則 | `firestore.rules` 檔案 | Firebase Console 別手動改（會跟本地版漂移）|
| 專案說明 | GitHub README | FB 貼文、YouTube 描述都連過來 |
| 進度與日誌 | 一個固定的工作筆記（Obsidian / Notion / Google 文件都行） | 不要三個地方各寫一份 |

> 💡 **以我自己的做法為例**：專案藍圖寫在 `CLAUDE.md`、進度日誌寫在 Obsidian 的 `專案工作流程.md`。你不一定要用 Obsidian，重點是**一個地方**、不要多處雙寫。

## 2. 開工前的六個關鍵問題

每次想做新工具，先在心裡（或對 Claude）回答這六題。**回答完，工具的雛形就出來了。**

### Q1：你想解決什麼問題？（一句話）

不要說「想做一個英文工具」。要說：
- ❌ 「我想做一個英文工具」
- ✅ 「我想讓學生在 5 分鐘內背完 10 個單字並馬上測驗，分數要能累積」

**越具體越好**，Claude 越知道要幫你做什麼。

### Q2：誰會用這個工具？

| 使用者 | 影響 |
|--------|------|
| 只有你自己 | 純前端就好，不用資料庫 |
| 你的學生（一個班） | 可能要 Firebase（要記住成績） |
| 全校 / 跨班 | 一定要 Firebase + 區分班級 |
| 公開給網路上其他老師 | 一定要 Firebase + 公開展示頁 |

### Q3：需不需要「記住」資料？

| 情境 | 要不要資料庫？ |
|------|-------------|
| 學生關掉網頁就不需要再看到內容（例：抽籤器、計算器） | ❌ 不需要 → 只用 GitHub |
| 學生提交的資料要給其他人看到（例：文字雲、回饋牆） | ✅ 需要 → 加 Firebase |
| 要累積學生長期表現（例：這學期答對率） | ✅ 需要 → 加 Firebase |
| 老師要事後分析全班結果 | ✅ 需要 → 加 Firebase |

> **判斷口訣**：重新整理瀏覽器，東西會不會消失？會消失但你不希望它消失 → 要 Firebase。

### Q4：老師要怎麼查資料？

**強烈建議：不要做老師登入網頁。**

直接用「Claude + Firebase MCP」用中文查就好：
- 「幫我查今天 715 班的回饋有哪些」
- 「列出最熱門的 5 個關鍵字」
- 「找出總分前三名的座號」

**好處**：少寫一個老師頁面、不用管登入、查詢方式無限彈性。
**前提**：老師端是「你自己」一個人，學生不會用到。

### Q5：學生身分怎麼處理？

🔒 **去識別化原則：絕對不要在資料庫存學生真名。**

| ✅ 可以存 | ❌ 不要存 |
|---------|---------|
| 座號（1, 2, 3, ...） | 學生真名 |
| 班級代號（715, 809） | 學生身分證字號 |
| 隨機 token（s01-592q） | 學生家長電話 |

**為什麼？** 因為資料會經過 AI 服務（Claude / Groq / Gemini）做分析。學生個資進到 AI 的訓練資料是無法回頭的事。

座號夠你認出是誰，AI 不知道是哪個小孩，這樣最安全。

> 🇹🇼 **台灣老師特別提醒**：以上是通用隱私原則。台灣《個人資料保護法》對教育資料有各縣市與學校層級的細則（例：有些學校規定學生資料不得上傳境外雲端服務）。**遇到衝突，以所在學校的個資管理規範為準**，不要覺得「技術上做得到就可以做」。

### Q6：這個工具你會用多久？（生命週期）

這題決定**架構該多「輕」還是多「重」**。

| 生命週期 | 你會用它多久 | 建議架構 | 維護成本 |
|---------|------------|---------|---------|
| 🎈 **一次性** | 一堂課用完就算了（抽籤器、暖身計時器） | 純前端 HTML，甚至不用 push 到 GitHub Pages | 零 |
| 📆 **單元性** | 這個單元（約 2-3 週） | 純前端；或 Firebase 但資料用完就刪 | 低 |
| 📚 **學期性** | 整學期都在用（課堂回饋牆、文字雲、作業繳交） | GitHub Pages + Firebase | 中（期末要清資料） |
| 🏫 **跨學期 / 跨年度** | 多個學期累積（成績歷程、長期學習軌跡） | GitHub Pages + Firebase + 班級/學年分流設計 | 高（要設計資料分區、考慮升級、升學時怎麼處理） |

**判斷口訣**：
- 🎈 一次性 → **別過度設計**，純前端就好，花 15 分鐘做出來用完丟掉
- 📚 學期性以上 → **從第一天就想好期末怎麼歸檔**（例：文字雲集合命名加日期 `wordcloud_2026spring`）

**最常見的誤判**：老師做「這學期回饋牆」時沒想過「下學期要不要延續」，結果變成每學期 commit 同一份代碼改集合名——應該第一天就把班級/學期寫進集合名或 document 欄位。

## 3. 三種專案範式（你要做哪一種？）

回答完上面六題，你會落在這三種範式之一：

### 範式 A：純前端工具（最簡單）

**情境**：學生用完即丟，不需要記住任何東西。

| 例子 | 說明 |
|------|------|
| 互動視覺化 | 拖拉看圖形變化 |
| 計算器 / 換算器 | 輸入數字算結果 |
| 抽籤器 / 計時器 | 課堂工具 |
| 單機小遊戲 | 練習用 |

**需要的工具**：只要 GitHub Pages。

**檔案數**：通常 1 個 `index.html`。

---

### 範式 B：學生提交 + 公開展示（最常用）

**情境**：學生輸入東西、其他人看得到。

| 例子 | 學生做的事 | 公開展示的東西 |
|------|----------|--------------|
| 即時文字雲 | 輸入詞彙 | 全班的詞變大字雲 |
| 課堂回饋牆 | 寫一句感想 | 顯示所有人的便利貼 |
| IRS 投票 | 選 A/B/C/D | 顯示長條圖 |
| 心情打卡 | 選表情 | 顯示班級今日氛圍 |
| 許願池 | 寫願望 | 公開展示 |

**需要的工具**：GitHub Pages + Firebase（Firestore）。

**檔案數**：通常 2 個 HTML
- `submit.html`（學生提交頁，QR Code 給學生掃）
- `display.html`（公開展示頁，投影出去給全班看）

**老師端**：Claude + Firebase MCP，用中文查。

---

### 範式 C：學生個人化（最強大）

**情境**：每個學生看到的東西不一樣，老師要看到每個人的進度。

| 例子 | 一人一碼 | 老師儀表板 |
|------|---------|-----------|
| 教學駕駛艙 | 學生掃自己的 QR Code | 即時看誰答對誰答錯 |
| 個人化練習 | AI 依學生程度出題 | 看每個學生卡在哪 |
| 差異化派題 | 學生收到專屬題目 | 即時派題給特定組別 |

**需要的工具**：GitHub Pages + Firebase（Firestore + 一人一碼 token 機制）。

**檔案數**：通常 3 個 HTML
- `index.html`（學生網頁，掃 QR Code 進入會帶 token）
- `qr-generator.html`（QR Code 總表，一次列印全班）
- `teacher-dashboard.html`（老師端即時儀表板）

> 💡 **這就是 EP11 教學駕駛艙在做的事**。是大魔王關，建議先把範式 B 玩熟再來。

## 🚨 3.5 暫停一下：這套做法什麼時候**不該用**？

> 這是 2026 年最常被忽略但最重要的一節。看完這節你才知道這套做法的**邊界**在哪。
>
> 你跟 AI 合作寫出來的東西，在業界有個新名詞叫 **vibe coding**（憑感覺寫程式）——[Andrej Karpathy 提出](https://brittanywashburn.com/2026/02/what-is-vibe-coding-for-teachers/)。
> 這種方式很適合老師做小工具，但**不是萬靈丹**，近一年業界開始累積不少警訊。

### ⚠️ vibe coding 的已知風險（近一年的業界數據）

| 風險 | 數據 | 來源 |
|------|------|------|
| AI 生成代碼含 OWASP Top 10 漏洞 | 45% | [Veracode](https://labs.cloudsecurityalliance.org/research/csa-research-note-ai-generated-code-vulnerability-surge-2026/) |
| vibe-coded 專案平均測試覆蓋率 | 12%（傳統開發 68%） | [The Server Side](https://www.theserverside.com/tip/The-case-against-vibe-coding) |
| 預估 2028 年被取消或重做的比例 | 40% | [Gartner / Beam](https://getbeam.dev/blog/ai-technical-debt-vibe-coding.html) |

**但你別緊張**——這些數據主要針對**企業級、長期維護、多人協作**的情境。**你做班級工具基本不會中標**，因為：
- 規模小（一個班 30-60 人）
- 生命週期短（一學期～一年）
- 資料去識別化（座號）
- 老師本人維護，壞了立刻找 Claude 修

### ✅ 這套做法**適合**什麼

- 一個班 / 一個年級的教學工具（≤ 100 人）
- 一學期～一學年生命週期的工具
- 資料可去識別化（只用座號、班級代號）
- 你自己是主要維護者，壞了能立刻找 Claude 修
- 錯了頂多重做，不會有學生權益損失或法律責任

### ❌ 這套做法**不適合**什麼

| 場景 | 為什麼不適合 | 建議做法 |
|------|------------|---------|
| 🚫 替代學校正式選課 / 成績系統 | 涉及官方成績、需稽核、出錯有法律責任 | 用學校既有系統，或申請資訊組正式開發 |
| 🚫 涉及金流（校外教學收費、班費管理） | 金流需要法遵、PCI-DSS 認證 | 用 Line Pay、信託帳戶、校務系統付款 |
| 🚫 需要官方存證的成績、出缺勤 | 必須有稽核軌跡、不得遺失 | 校務系統 |
| 🚫 跨校 / 跨縣市協作平台（1000+ 老師用） | 技術債會被放大、你一個人修不完 | 申請正式開發經費、找工程團隊 |
| 🚫 儲存學生真名、身分證字號、家長電話 | 個資外洩風險；若用 AI 分析會進訓練資料 | 去識別化到只剩座號；敏感資料留在校務系統 |
| 🚫 一週內要發給全校、不容失敗 | AI 生成代碼需時間驗證、邊緣情況易出錯 | 找既有成熟工具，不要自建 |

### 🧭 判斷簡表：這個想法該不該做？

快速自我檢查，任一項打 ❌ 就該暫停：

- [ ] 最多影響多少人？（>100 人要三思）
- [ ] 出錯的後果嚴重嗎？（影響成績、金錢、隱私 → 不做）
- [ ] 我是唯一維護者嗎？（超過兩個老師協作 → 要升級架構）
- [ ] 資料能不能去識別化？（不能 → 不做或用校務系統）
- [ ] 壞了能立刻修嗎？（半夜要交報告壞掉沒人修 → 不做）
- [ ] 這件事有現成工具嗎？（有 → 先用現成的）

### 💡 結論

這套做法**是教學工具的最佳起點，不是所有事情的終點**。
認識邊界，你才能在邊界內做得安心；越界的事情，該找工程團隊就找工程團隊。

## 4. 標準起手式（每個專案都一樣的 6 步）

不管你做哪一種範式，前 6 步都一樣：

### Step 1：在 Google 雲端硬碟開新資料夾

直接在 `G:\我的雲端硬碟\` 底下開一個新資料夾，**用英文 + 數字命名**最好（中文也行但 GitHub repo 名後面要轉英文）：

```
G:\我的雲端硬碟\my-vocab-game\
```

### Step 2：用 Claude Code 桌面版打開這個資料夾

「新增專案」→ 選剛開的那個資料夾。

### Step 3：對 Claude 說「請幫我做專案起手式」

Claude 會自動跑完這幾步：
- 建 `CLAUDE.md`（專案說明書）
- 建 `.gitignore`（排除敏感檔）
- `git init` + 第一次 commit
- 建 GitHub **公開** repo（要用 GitHub Pages 必須公開）並 push
- 在 Obsidian vault 建對應的專案資料夾 + 一份「工作筆記.md」（如果你裝了 Obsidian MCP）

第一次做的時候每一步 Claude 會跟你確認，做過 3-5 次熟了可以加一句「全部跑下去不用問我」加速。

> 💡 **進階選配**：看過 EP14 的老師可以設一個「全域初始化 SOP」，讓 Claude 每次遇到未初始化的 GDrive 資料夾就自動提醒你。一般老師不用那樣做，每個新專案直接打「請幫我做專案起手式」就好。

### Step 4：寫第一版 HTML（前端）

對 Claude 講你想做什麼，越具體越好。範例見下面第 5 節的句型範本。

### Step 5：視需求接 Firebase（如果是範式 B 或 C）

對 Claude 說：「請幫我接 Firebase」。Claude 會自動：
- 用 Firebase MCP 建（或選）一個 Firebase 專案
- 建 Firestore 資料庫
- 寫 `firestore.rules`（白名單模式）
- `npx firebase-tools deploy --only firestore:rules` 部署規則
- 在 HTML 裡加 `firebaseConfig` 跟讀寫程式碼

### Step 6：push 到 GitHub Pages 上線

對 Claude 說：「請 push 並開啟 GitHub Pages」。
1 分鐘後拿到網址：`https://你的帳號.github.io/repo名/`

> 🟡 **要點**：GitHub Pages 免費版只支援**公開 repo**。如果初始化時建的是私有 repo，要先到 GitHub 設定改公開。

## 6. 必踩的坑（前期就要知道）

### GitHub 篇

| 坑 | 解法 |
|----|------|
| 私有 repo 不能用 GitHub Pages | 改公開 repo，或付費升級 |
| 單一檔案超過 100 MB 推不上去 | 大檔（影片）放 YouTube，網頁只放嵌入連結 |
| 在 Google 雲端硬碟資料夾 `git commit` 報錯：`cannot update the ref` | 設一次：`git config windows.appendAtomically false`（每個 GDrive 內 repo 設一次） |
| 中文資料夾名變 GitHub repo 後 URL 很醜 | 資料夾用英文，或建 repo 時另取英文名 |

### Firebase 篇

| 坑 | 解法 |
|----|------|
| 專案名 `my-tool` 變成 `my-tool-a3b9c`（自動加後綴） | 建立後到 Project Settings 確認真正的 Project ID，`.firebaserc` 要寫加後綴版 |
| Firestore 規則選「測試模式」30 天就過期 | 一律選**「正式版模式」+ 立刻寫白名單** |
| `firebaseConfig` 看起來很像密碼，能公開嗎？ | **可以公開**，apiKey 和 projectId 設計給前端用，不算敏感 |
| Firebase MCP 工具找不到 | 在對話裡實際叫一次 firestore 工具觸發 ToolSearch；或關閉重開 Claude Code |
| `firestore_query_collection` 報「Collection id is invalid」 | `collection_path` 不要含尾巴 `/`，寫 `wordcloud_words` 不要 `wordcloud_words/` |
| `firebase login` 在 Claude Code 對話裡卡住 | 必須打開 cmd / PowerShell 互動式終端執行，無法在對話中完成 |

### Obsidian 篇

| 坑 | 解法 |
|----|------|
| 雙寫漂移：進度寫三個地方 | 遵守 SSOT 原則，只在 `專案工作流程.md` 寫進度，其他地方都用連結指過來 |
| Obsidian MCP 工具預設沒載入 | 跟 Firebase MCP 同樣要 ToolSearch 觸發；讓 Claude 實際呼叫一次 |
| 把學生成績寫在 Obsidian | 該放 Firebase，Obsidian 只寫「為什麼這樣設計成績制度」的決策 |

### 安全 / 個資篇

| 坑 | 解法 |
|----|------|
| 學生真名進到 AI 訓練資料 | 一律只存座號 + 班級代號 |
| API Key 寫死在 HTML 被人複製 | Firebase Config 沒關係（公開的）；但 OpenAI / Groq / Gemini key 不要直接寫前端 |
| Firestore Rules 預設開放任何人讀寫 | **馬上**改白名單：只開你要的集合，其他全擋 |
| `.env` 不小心 commit 推上去 | `.gitignore` 一定要把 `.env`、`*.key`、`credentials.*` 列進去 |

## 7. 完成後的檢查清單

每做完一個工具，照這個清單確認一次：

### GitHub 端
- [ ] 公開展示頁網址打開沒問題（用無痕視窗測）
- [ ] `.gitignore` 有排除敏感檔案
- [ ] commit 訊息可讀，不是「更新」兩個字
- [ ] GitHub README 說明這個工具是做什麼的

### Firebase 端
- [ ] 用三個不同裝置同時提交，資料都正確進 Firestore
- [ ] 對 Claude 說「請查 YYY 集合的內容」能拿到資料
- [ ] Firestore Rules 不是全開的（試著用無痕視窗從 console 打 API 應該被擋）
- [ ] 沒有把學生真名 / 個資寫進資料庫

### Obsidian 端
- [ ] 今天改了什麼、為什麼這樣改，記一筆到專案的工作筆記
- [ ] 踩到的新坑寫下來，下次不要重踩同個坑
- [ ] 這個專案在 vault 有自己的資料夾，裡面至少有一份「工作筆記.md」

（不用 Obsidian 的老師改用 Notion / Google 文件，概念一樣——每個專案一個固定地方記工作筆記，不要三個地方各寫一份。）

## 8. 當你卡住時

### 「Claude 一直做不出我想要的東西」
- 把問題拆得更細：不要說「做一個英文工具」，要說「做一個 5 題單字選擇題的網頁，每題 4 個選項，答完顯示分數」
- 截圖丟給 Claude 看你想要的長相
- 直接拿你看過的網頁網址當參考：「請參考 https://XXX 的設計」

### 「Firebase 連不上」
- 對 Claude 說：「請查 firestore.rules 跟 HTML 裡的 firebaseConfig 對不對得上」
- 確認 Project ID 是不是被自動加後綴的版本

### 「網頁上線了但學生看到的跟我看到的不一樣」
- 學生看的是 GitHub 上的版本，你電腦上的修改如果沒 push 還沒上線
- 對 Claude 說：「請把最新版本 push 到 GitHub」

### 「我忘記上次這個功能是怎麼做的」

兩條並行的路，看哪個你當下比較快：

**路 1：從 GitHub commit history 找**（「**做了什麼**」層面）
- 打開 GitHub repo 的 commits 頁，找關鍵字
- 或對 Claude 說：「請看 git log，幫我找『即時派題』這個功能是哪個 commit 做的」
- 前提：你的 commit 訊息有寫清楚（第 1 節為什麼強調紀律就是這原因）

**路 2：從 Obsidian 工作筆記找**（「**為什麼這樣設計**」層面）
- 打開專案資料夾的「工作筆記.md」，看「最近更動紀錄」或「踩坑筆記」
- GitHub 記「做了什麼」、Obsidian 記「為什麼」
- 兩個合起來才是完整故事

### 「我想要的功能其他人有做過嗎」
- 看本 repo 的範例：[2026database](https://github.com/mathruffian-dot/2026database)
- 看一人一碼版本：[math-cockpit](https://github.com/mathruffian-dot/math-cockpit)
- 看 Claude 基本功 EP09 / EP09.5 / EP11 的影片

### 🆘 「我做壞了，想回到上一個可用版本」（回退機制）

**這節是救命稻草**。Claude 有時會把一個可用的版本改壞，這時候不要慌，三個家各有自己的「回退」方式：

#### 🐙 GitHub 端：程式碼回退

```
請幫我看 git log，找出最近一次「XXX 功能還能用」的 commit，把那個版本 checkout 出來讓我預覽。
確認沒問題的話，再幫我 revert 到那個 commit 並 push。
```

進階版（只退一個檔案）：
```
請幫我把 index.html 恢復到上個 commit 的版本，其他檔案不動。
```

💡 **前提**：commit 訊息要寫得好，不然你分不出哪個版本「還能用」。再次印證為什麼第 1 節強調 commit 訊息紀律。

#### 🔥 Firebase 端：資料回退

Firebase **Spark 免費方案沒有自動備份**（Blaze 方案才有）。所以：

```
請把 YYY 集合裡，created_at 晚於 2026-04-15 00:00:00 的所有文件列出來給我確認，
確認沒問題的話幫我刪除這些髒資料。
```

**預防優於治療**：
- 做大改動前，先對 Claude 說：「請把 YYY 集合目前所有文件匯出成 JSON 給我存成本地檔案」
- 月末順手匯出一次，當作手動備份

#### 📘 Obsidian 端：筆記回退

Obsidian 內建 **Settings → File recovery**（預設每 5 分鐘自動快照）。點開就能回到幾小時前、幾天前的版本。

進階（如果你的 vault 也有進 Git），可以請 Claude 幫你查：

```
請幫我看 vault 的 git 歷史，找出「<筆記名>.md」三天前的版本給我看。
```

#### 🛡️ 最壞情況：整個 repo 復活

```
請從 GitHub 最新成功部署的版本重新 clone 一份下來覆蓋我的本地資料夾。
```

> ⚠️ 這個操作會覆蓋你本地所有未 push 的變更，做之前務必確認你沒有「還沒 push 但想留的修改」。

## 附錄 A：進階自動化（給已上手的老師）

> ⚠️ **對第一次做專案的老師**：本附錄是給做過幾個專案、覺得「每次收工都要記得 commit」很煩的老師看的。初學者**完全可以跳過**，先把 Lv.1-2 的工具做滿幾個再回頭看。
>
> **只有一台電腦的老師**：只需要看 **A.1～A.3**（收工自動化本身）。A.4（多電腦）可以跳過。

做了幾個專案後，你會發現「每次收工要記得 commit + push、更新筆記」很麻煩。Claude Code 有兩個進階機制可以讓收工半自動或全自動。

### A.1 「自動」其實有三個層級

| 層級 | 設在哪 | 誰執行 | 強度 |
|------|-------|--------|-----|
| **規則** | `CLAUDE.md` 文字敘述 | Claude（讀到、自己決定） | 🟡 軟性，可能漏 |
| **Skill（`/收工`）** | `~/.claude-skills/shutdown/SKILL.md` | Claude（你打指令才觸發） | 🟠 半自動 |
| **Hook（SessionEnd）** | `~/.claude/settings.json` | Claude Code 桌面版本身 | 🟢 硬性，100% 跑 |

**口訣**：CLAUDE.md = 請求、Skill = 你打指令、Hook = 自動。

### A.2 推薦：兩個都設（雙保險）

#### 方案 1：建 `/收工` Skill（智能版）

在 `~/.claude-skills/shutdown/SKILL.md` 寫一份 SOP，內容大致：
- LLM 摘要今天做了什麼
- 更新 Obsidian 駕駛艙日誌「最近更動紀錄」
- 智能 commit message（含今天工作重點）+ push

收工時對 Claude 說「**收工**」或「**該同步的同步**」即可一鍵執行。

#### 方案 2：SessionEnd Hook（無腦備份版）

在 `~/.claude/settings.json` 加：

```json
"hooks": {
  "SessionEnd": [
    {
      "hooks": [
        {
          "type": "command",
          "command": "bash ~/.claude/scripts/session-cleanup.sh"
        }
      ]
    }
  ]
}
```

腳本 `session-cleanup.sh` 保守版核心邏輯：
- 只 `git add -u`（tracked 且 modified）
- **不**自動 add 新檔（避免誤 commit 測試檔）
- commit message 只寫時間、不做摘要
- push 後寫 log 到 `~/.claude/scripts/session-cleanup.log`

### A.3 Skill vs Hook 怎麼分工

| 情境 | 誰接手 | 結果 |
|------|-------|------|
| 你主動打「收工」 | **Skill** | 智能摘要 + 更新駕駛艙 + 寫好 commit |
| 你忘記打、直接關 Claude Code | **Hook** | 純 bash 腳本做 git commit + push 備份 |

兩個**不會衝突**：Skill 跑完後，Hook 看到沒有 staged 變動就會退出。

### A.4 有多台電腦怎麼辦？

**95% 的老師只有一台電腦，這節可以直接跳過。**

如果你剛好學校、家裡各一台都在工作，Skill 跟 Hook 的跨電腦同步機制請看 **Claude 基本功 EP14**（跨電腦工作術）——背後需要 chezmoi + dotfiles git repo 這套基礎建設，不是一個附錄段落能講完的。

### A.5 什麼時候先不要開 Hook？

- 還沒熟 git 的時候（突然多了 commit 會慌）
- 想保留「手動才 commit」的紀律
- 專案在 PoC 階段，測試檔多、不想被 auto-commit

### A.6 驗證有沒有生效

1. **Skill 生效驗證**：對 Claude 說「收工」，看它有沒有主動執行更新駕駛艙 + commit 流程
2. **Hook 生效驗證**：隨便改一個 tracked 檔、關 Claude Code，檢查 `~/.claude/scripts/session-cleanup.log` 有沒有新紀錄，以及 GitHub 有沒有多一個 `[SessionEnd 自動保存]` commit

> 💡 **實作細節可以直接看我的範例**：
> - Skill：`~/.claude-skills/shutdown/SKILL.md`
> - 腳本：`~/.claude/scripts/session-cleanup.sh`
> - 詳細原理會在 **Claude 基本功 EP14（規劃中）** 影片完整示範
